# Тестове завдання: Invoice Management System

## Інструкція зі запуску

### 1. Зібрати образи та запустити контейнери у фоні
```bash
docker-compose up -d --build
```

### 2. Створити структуру бази даних (міграції)
```bash
docker exec -it invoice_api php artisan migrate
```

### 3. Доступ до сервісів
- **Frontend (Nuxt)**: http://localhost:3001
- **Backend API (Laravel)**: http://localhost:8001/api/invoices

---

## 1. Структура frontend та backend

### Frontend (`./frontend/`)
- **Фреймворк**: Nuxt 4 (Vue 3) з SSR (Server‑Side Rendering) та TypeScript.
- **Архітектура**:
  - `app/pages/` – сторінки (`index.vue` для списку, `[id].vue` для деталей).
  - `app/components/` – перевикористовувані компоненти (таблиця, форма).
  - `app/composables/` – логіка роботи з API (`useInvoices.ts`).
  - `app/utils/` – допоміжні функції (форматування, валідація).
  - `app/types/` – TypeScript‑інтерфейси.
- **UI‑бібліотека**: Nuxt UI (на базі Tailwind CSS).
- **Конфігурація**: `nuxt.config.ts` із розділенням runtime‑конфігурації для серверних та клієнтських запитів.
- **Збірка**: Docker‑образ на основі `node:20‑alpine` із монтуванням вихідних кодів для hot‑reload у розробці.

### Backend (`./backend/`)
- **Фреймворк**: Laravel 12 (PHP 8.4).
- **Архітектура**:
  - RESTful API (`routes/api.php` → `InvoiceController`).
  - Eloquent‑моделі, міграції, фабрики.
  - Базова валідація в контролері.
- **База даних**: MySQL 8.0 (контейнер `mysql:8.0`).
- **Конфігурація**:
  - `.env` із параметрами підключення до БД.
  - Middleware `HandleCors` увімкнений за замовчуванням (дозволяє запити всередині Docker‑мережі).
- **Збірка**: Docker‑образ на основі `php:8.4‑fpm` з artisan serve.

### Інфраструктура (`docker‑compose.yml`)
- Три сервіси: `db` (MySQL), `backend` (Laravel), `frontend` (Nuxt).
- Мережа `invoice_network` для внутрішньої комунікації.
- Змінні оточення для розділення URL:
  - `NUXT_API_URL=http://backend:8000/api` – для SSR‑запитів (всередині мережі).
  - `NUXT_PUBLIC_API_BASE=http://localhost:8001/api` – для клієнтських запитів (браузер).
- Том `mysql_data` для збереження даних БД.

## 2. Компроміси та їх причини

1. **Режим розробки фронтенда** (`npm run dev` із монтуванням томів)  
   **Причина**: Дозволяє миттєво застосовувати зміни коду без перезбірки образу, що критично при активній розробці та відладці.  
   **Компроміс**: Не є production‑режимом (менша продуктивність, відсутня статична оптимізація).

2. **Один Docker‑образ для бекенда** (php:8.4‑fpm + artisan serve)  
   **Причина**: Простота налаштування та запуску. Для тестового завдання достатньо вбудованого PHP‑сервера.  
   **Компроміс**: У продакшені потрібен Nginx/Apache для статики, балансування та безпеки.

3. **Очищення тому MySQL при збої ініціалізації**  
   **Причина**: На диску контейнера закінчилося місце, що заважало створенню файлу `auto.cnf`. Без очищення контейнер потрапляв у безкінечний цикл перезапусків.  
   **Компроміс**: Втрата даних (прийнятно для тестового середовища, але не для продакшену).

4. **CORS через Docker‑мережу, без явної конфігурації в Laravel**  
   **Причина**: Всередині Docker‑мережі всі сервіси мають спільний origin, тому CORS‑політика не блокує SSR‑запити. Клієнтські запити йдуть з localhost:3001 на localhost:8001 (той самий origin для браузера).  
   **Компроміс**: При розгортанні на різних доменах знадобиться явне налаштування CORS у `config/cors.php`.

5. **Базові валідації в контролері**  
   **Причина**: Для демонстрації достатньо вбудованих методів Laravel.  
   **Компроміс**: У реальному проекті потрібні Form‑Request класи, суворіші правила та кастомні повідомлення про помилки.

## 3. Що покращити у production‑версії

1. **Фронтенд**:
   - Перейти на production‑збірку (`npm run build`), прибрати монтування томів.
   - Налаштувати CDN для статики (з `.output/public`).
   - Додати code‑splitting, оптимізацію зображень.
   - Впровадити серверний кеш для SSR‑сторінок (Redis).

2. **Бекенд**:
   - Замінити `artisan serve` на зв’язку Nginx + PHP‑FPM.
   - Налаштувати черги для фонових задач (наприклад, генерація PDF‑рахунків‑фактур).
   - Додати повноцінну аутентифікацію (Sanctum) та авторизацію (ролі).
   - Реалізувати пагінацію з фільтрацією (за статусом, датою, сумою).
   - Впровадити версіонування API.

3. **Інфраструктура**:
   - Додати healthchecks для всіх сервісів у `docker‑compose.yml`.
   - Налаштувати резервне копіювання тому MySQL (наприклад, через cron).
   - Використовувати `.env.production` із секретами (не зберігати в репозиторії).
   - Додати моніторинг (Prometheus + Grafana) та логування (ELK‑стек).

4. **Безпека**:
   - Явна конфігурація CORS із зазначенням дозволених origins.
   - Rate‑limiting для API (наприклад, `laravel‑rate‑limiter`).
   - Валідація всіх вхідних даних на стороні сервера.
   - Захист від SQL‑ін’єкцій (Eloquent використовує prepared statements, але варто перевірити сирі запити).

5. **UX/UI**:
   - Додати loading‑скелетони замість спінерів.
   - Реалізувати infinite‑scroll або віртуалізацію для великих списків.
   - Кешувати дані на клієнті (наприклад, через `useAsyncData` з ключами).
   - Обробляти помилки мережі та показувати user‑friendly повідомлення.

## 4. Враховувані UX edge cases

1. **Розділення URL для SSR та клієнтських запитів**  
   Проблема: SSR‑запити з контейнера фронтенда не могли достукатися до бекенда за `localhost:8001`.  
   Рішення: У `useInvoices` додана перевірка `process.server ? config.apiUrl : config.public.apiBase`.

2. **Скидання стану форми при закритті модалки**  
   Проблема: Після редагування інвойса модалка могла відкритися в режимі редагування для нового інвойса.  
   Рішення: `watch(isOpen, ...)` скидає `selectedInvoice` при закритті модалки.

3. **Перестворення форми при зміні режиму (створення ↔ редагування)**  
   Проблема: Reactivity computed‑полів (`isEditing`) не завжди оновлювалася при зміні `initialData`.  
   Рішення: Додано `:key="selectedInvoice?.id || 'create'"` на компонент `InvoiceForm`.

4. **Захист від передачі події замість `null` у `openModal`**  
   Проблема: Кнопка «Створити інвойс» передавала об’єкт події (PointerEvent), що ламало логіку.  
   Рішення: Перевірка `if (invoice && typeof invoice === 'object' && 'isTrusted' in invoice)`.

5. **Блокування полів при статусі інвойса не «pending»**  
   Реалізовано: computed `isLocked` блокує поля сум та дати оплати, якщо інвойс уже оплачений або скасований.

6. **Валідація на стороні клієнта**  
   Використані Zod‑схеми (`invoiceCreateSchema`, `invoiceUpdateSchema`), які:
   - Показують помилки під полями.
   - Запобігають відправленню невалідних даних.
   - Мають різні правила для створення та оновлення.

7. **Зворотний зв’язок при збереженні**  
   - Кнопка відправлення переходить у стан `loading`.
   - При помилці API виводиться повідомлення в консоль (у production можна показувати toast‑сповіщення).
   - Після успішного збереження список інвойсів автоматично оновлюється.

8. **Доступність (a11y)**  
   - Усі форми мають семантичну розмітку (`<UForm>`, `<UFormGroup>`).
   - Поля пов’язані з мітками через `name`.
   - Кнопки мають зрозумілі label’и та стани (`disabled`, `loading`).
